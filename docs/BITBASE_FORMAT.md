# DRAUGHTS Endgame Bitbase — формат файла

> Документ фиксирует спецификацию формата эндшпильной базы DRAUGHTS.
> Цель — сделать формат пригодным для внешних интеграций и
> использования третьими лицами; в текущей индустрии русских шашек
> 8×8 публичного стандарта не существует (см. `.planning/product/DECISIONS.md`,
> D38), и этот документ — попытка зафиксировать один.
>
> **Версия спецификации:** 1.0 (2026-04-18)
> **Действительна для:** bitbase JSON файлов, генерируемых
> `python -m draughts.tools.build_bitbase` начиная с DRAUGHTS 4.x.

## 1. Назначение

Эндшпильная база (bitbase) — предвычисленная таблица точных исходов
(Win/Draw/Loss) для всех позиций с небольшим числом фигур. Движок
обращается к ней в эндшпиле для безошибочной игры, минуя поиск.

Формат, описанный здесь, хранит **только W/D/L метку** (без числа
полуходов до выигрыша, distance-to-conversion). Это упрощает файл и
держит размер в рамках JSON для N ≤ 4 фигур. Для более крупных баз
потребуется бинарный формат (не в рамках v1.0).

## 2. Правила игры

База применима **исключительно** к русским шашкам 8×8 (FMJD GameType
`25`):

- Доска 8×8, 32 тёмные клетки.
- Пешка ходит на одну клетку по диагонали вперёд.
- Пешка бьёт **вперёд и назад**.
- Дамка ходит и бьёт на **любое расстояние** по диагонали.
- Обязательное взятие (при нескольких вариантах — любое по выбору игрока).
- Пешка достигает последней горизонтали → становится дамкой; если это
  происходит во время серии взятий, она продолжает бить уже как дамка
  (исключение: «турецкий удар»).

Базы для **других вариантов шашек** (международные 10×10, английские
8×8 без взятия назад, бразильские, pool, чекеры и т.д.) семантически
несовместимы: значения W/D/L рассчитаны при других наборах ходов.

## 3. Контейнер файла

| Расширение | Содержимое |
|---|---|
| `bitbase_N.json`    | JSON-объект, UTF-8, как описано ниже |
| `bitbase_N.json.gz` | gzip того же JSON |

Префикс `N` — максимальное количество фигур, покрытых базой
(например, `bitbase_3.json` содержит все 1-, 2- и 3-фигурные позиции;
`bitbase_4.json` добавляет 4-фигурные, и т.д.).

## 4. Структура JSON

```
{
  "<zobrist_hash>": <wdl_int>,
  ...
}
```

- Ключ — десятичное представление 64-битного целого числа Zobrist-хэша
  позиции (см. §5), обёрнутое в строку (JSON не поддерживает целые
  числа произвольной длины в ключах).
- Значение — целое число:
  - `1` — WIN (сторона, ход которой, выигрывает при лучшей игре обеих сторон)
  - `0` — DRAW
  - `-1` — LOSS

Пример (фрагмент):

```json
{
  "1234567890123456789": 1,
  "9876543210987654321": 0,
  "1111222233334444555": -1
}
```

Пустой файл `{}` допустим (пустая база).

## 5. Zobrist-хэш

Ключ строится следующим образом:

1. Для каждой заполненной клетки `(y, x)` с фигурой значения
   `piece ∈ {1, 2, -1, -2}` (см. внутреннюю кодировку в
   `draughts.config`) вычисляется индекс типа фигуры:
   - `0: empty` (не используется)
   - `1: чёрная пешка (BLACK)`
   - `2: чёрная дамка (BLACK_KING)`
   - `3: белая пешка (WHITE)` ← `piece == -1`
   - `4: белая дамка (WHITE_KING)` ← `piece == -2`
2. Хэш позиции = XOR по всем заполненным клеткам значений из
   **псевдослучайной таблицы** `Z[y, x, piece_index]` размера
   `8×8×5 × int64`.
3. Если ход чёрных — XOR с константой `Z_SIDE` (отдельное 64-битное
   случайное число).

Таблица `Z` и `Z_SIDE` генерируются детерминистически через
`numpy.random.RandomState(seed=0xDEAD_BEEF)` с единицами в диапазоне
`[1, 2^63)`. Это даёт одинаковые хэши между запусками и между
разными инсталляциями DRAUGHTS.

Любая внешняя реализация должна воспроизводить **ровно ту же таблицу
и ту же функцию XOR**, иначе файл окажется нечитаемым.
Референсная реализация — `draughts/game/ai/tt.py::_zobrist_hash`.

### 5.1 Почему Zobrist, а не perfect-hash

Текущая спецификация использует Zobrist ради простоты. Промышленные
bitbase-форматы (Kingsrow, Scan) используют **perfect-hash** (ranking
function) — позиция → уникальный индекс 0..K-1, где K = точное число
допустимых расстановок. Это даёт ~2-3× более компактное хранение и
нулевую вероятность коллизий, ценой более сложного кода индексации.

Рефакторинг на perfect-hash рассматривается для версии 2.0 формата
(см. DECISIONS.md § D38). Версия 1.0 использует Zobrist сознательно.

## 6. Какие позиции в базе

Для `bitbase_N.json` перечисляются все позиции, удовлетворяющие всем
условиям:

- Количество фигур на доске ≤ N.
- Фигуры стоят только на тёмных клетках (`x % 2 != y % 2`).
- Пешки не находятся на своём ряду превращения (они уже стали бы
  дамками — такие позиции недостижимы).
- Обе стороны могут иметь фигуры (в т.ч. 0 у одной из сторон, что
  соответствует уже выигранной позиции).
- Обе стороны-то-move (для каждой расстановки фигур хранятся две
  записи — одна для белого хода, одна для чёрного — с разными
  Zobrist-хэшами, отличающимися на `Z_SIDE`).

Позиции вне этого множества в файле не хранятся. Запрос неизвестного
хэша возвращает «вне базы», и движок обращается к обычному поиску.

## 7. Как пользоваться программно

### Python (рефренс-реализация)

```python
from draughts.game.ai.bitbase import EndgameBitbase, WIN, DRAW, LOSS

# Загрузить (автоматически распознаёт .gz)
bb = EndgameBitbase.load("path/to/bitbase_4.json.gz")

# Запросить позицию: compute Zobrist → lookup
from draughts.game.ai.tt import _zobrist_hash
from draughts.game.board import Board
from draughts.config import Color

board = Board(...)  # your position
h = _zobrist_hash(board.grid, Color.WHITE)
result = bb.probe(h)  # -> WIN (1) | DRAW (0) | LOSS (-1) | None (не в базе)
```

### Генерация собственного файла

```bash
python -m draughts.tools.build_bitbase \
    --max-pieces 4 \
    --output my_bitbase_4.json
```

Занимает часы и несколько гигабайт RAM для N=4; не предназначено для
запуска в интерактивном GUI (решение D36).

## 8. Контроль целостности

Файлы, распространяемые через GitHub Releases, сопровождаются
`<filename>.sha256` — текстовым файлом с hex-дигестом SHA-256. Формат:

```
<64 hex chars>  bitbase_4.json.gz
```

Первый токен (64 символа) — дайджест. Downloader DRAUGHTS
(`draughts.tools.bitbase_downloader`) проверяет дайджест после
скачивания и отвергает файл при несовпадении.

## 9. Размеры (ориентировочно, v1.0)

| N | Позиций | JSON (raw) | JSON (gz) | Время генерации |
|---|---:|---:|---:|---|
| 3 | 399 280    | ~14 MB   | ~3 MB     | ~2 мин       |
| 4 | 12 817 912 | ~300 MB  | ~126 MB   | ~4-5 часов   |
| 5 | ~400 млн   | ~15 GB   | ~5-6 GB   | дни, не рекомендуется в JSON |

5-piece и выше целесообразно переводить на бинарный формат
(perfect-hash + packed bits); это работа для v2.0 спецификации.

## 10. Обратная / прямая совместимость

- Будущие версии (1.x) могут добавить **опциональные поля** к value
  (распаковка из `int` в объект `{"wdl": 1, "dtc": 15}` для
  distance-to-conversion). Читатели v1.0 должны игнорировать
  неизвестные поля.
- Несовместимые изменения (например, смена Zobrist seed или
  переход на perfect-hash) попадут в v2.0 с обязательным bump
  префикса имени файла (`bitbase_v2_4.json`).
