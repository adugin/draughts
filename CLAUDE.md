# DRAUGHTS — контекст проекта для Claude

## Цель проекта

Русские шашки на **Python 3.12+ / PyQt6 / NumPy**. Цель —
**продукт мирового класса**, совмещающий лучшие фичи ведущих шашечных
программ (Kingsrow, Scan, Dam 3.0, CheckerBoard, Windraughts и др.).

Оригинал: Borland Pascal 7.0 (~1998–2000), полный разбор:
**[ANALYSIS.md](ANALYSIS.md)**. Отход от legacy разрешён везде, где это
даёт заметное улучшение; международные шашечные стандарты (FMJD, PDN)
имеют приоритет над наследственными решениями.

## Управление: Product Owner

**Владелец продукта — агент `gsd-new-project` / `general-purpose` в роли
Product Owner.** Он принимает решения о том, какие фичи добавлять, в
каком порядке, какие legacy-решения менять, какие стандарты внедрять.
Артефакты PO хранятся в `.planning/product/`:

- `.planning/product/ROADMAP.md` — упорядоченный список фич
- `.planning/product/DECISIONS.md` — принятые архитектурные решения
- `.planning/product/RESEARCH.md` — анализ конкурирующих продуктов

Перед любой существенной фичей — **консультация с PO**. PO анализирует
запрос в контексте roadmap, говорит "да/нет/сначала сделай X", обновляет
DECISIONS.md. Только после этого приступаем к реализации.

## Политика выбора модели

**Opus 4.6** — для стратегии, архитектуры, сложного поиска, рефакторинга
с рисками, анализа корневых багов. Дорого, но качественно.

**Sonnet 4.6** — для сбора данных, типовых тестов, линейного рефакторинга,
документации, механических правок. Большинство sub-agent работы — Sonnet.

**Haiku 4.5** — не используется (не хватает reasoning для этого проекта).

**Правило:** если задача описывается в 3 предложениях и имеет чёткие шаги
— Sonnet. Если требует принятия архитектурных решений или глубокого
анализа кода — Opus. Orchestrator (главный чат) — Opus, делегирует
Sonnet-агентам всё, что можно сформулировать как конкретное задание.

## Benchmark-driven development

**Правило:** любое изменение в `draughts/game/ai.py` должно быть
измерено A/B бенчмарком против предыдущей версии перед тем как
считаться улучшением. Инфраструктура:

- `.planning/bench.py` — trap battery + self-play blunder rate
- `.planning/head2head.py` — head-to-head (NEW vs OLD через importlib)
- `.planning/ai_old.py` — копия предыдущей версии для сравнения

**Workflow алгоритмических изменений:**
1. Перед правкой: `cp draughts/game/ai.py .planning/ai_old.py`
2. Внести изменение, прогнать `pytest`
3. Запустить `python .planning/head2head.py --games 60 --depth 5`
4. Если score NEW < 55% или CI пересекает 50% на 60 играх — изменение
   считается не улучшением. Либо откатывается, либо расширяется
   тестовая выборка до 100+ игр.
5. Если подтверждено улучшение — коммит с цифрами в commit message.

**Никаких "улучшений на глаз".** Только измеримые.

**Инвариант для dev-mode:** все тесты AI vs AI обязаны использовать
hard limits: `max_ply`, `move_timeout`, `game_timeout`, `quiet_move_limit`.
Никаких бесконечных циклов. Дефолты в `HeadlessGame.play_full_game` и
`Tournament` выставляются так, чтобы случайный запуск не мог подвесить
процесс больше чем на разумное время (см. коммит `3c101f4`).

## Международные стандарты

- **Нотация:** PDN (Portable Draughts Notation) для сохранений и экспорта
  партий. `GameType "25"` для русских 8×8. См. `draughts/game/pdn.py`.
- **Алгебраическая нотация** `a1..h8` на всех уровнях: UI, CLI,
  heartbeat логах, комментариях в коде.
- **Уровни сложности:** отображаются как приблизительный рейтинг
  (например, ~1200, ~1600, ~2000 Elo) с указанием глубины в скобках.
  Никаких "лёгкий/средний/сложный" без числовых ориентиров.
- **Локализация:** RU основная, но строки-источники в коде должны быть
  готовы к i18n (gettext-совместимый формат).

## Команды разработки

```bash
# Запуск игры
python main.py                          # новая игра
python main.py game.json                # загрузить партию
python main.py --resume                 # продолжить из автосохранения
python main.py --difficulty 3 --black   # сложность + играть чёрными
python main.py --depth 8               # ручная глубина AI

# Бенчмарк AI (глубина 1-8, профилирование)
python benchmark.py --depth 8 --profile

# Тесты (запускать перед коммитом)
python -m pytest tests/ -q

# Проверка типов (strict на core + engine)
mypy draughts/game/ draughts/engine/

# Линтер + форматтер (ruff)
ruff check draughts/ tests/
ruff check --fix draughts/ tests/
ruff format draughts/ tests/
```

Конфигурация ruff и mypy — в `pyproject.toml`.

## Архитектура

- **Доска**: `np.ndarray((8,8), dtype=np.int8)`, 0-индексация, знаковая кодировка: `BLACK=1, BLACK_KING=2, WHITE=-1, WHITE_KING=-2, EMPTY=0`
- **AI**: Alpha-beta minimax + транспозиционная таблица (Zobrist), iterative deepening, quiescence search, LMR, killer moves
- **UI**: минималистичный — только доска + стандартное меню Windows
- **Фреймворк**: PyQt6 + NumPy
- **Язык интерфейса**: только русский
- **Сохранения**: JSON
- **Автор**: Andrey Dugin — в «О программе»

---

## Ключевые факты о коде

- Тёмные клетки: `x % 2 != y % 2` → 32 активные клетки
- `y=0` — верх (чёрные), `y=7` — низ (белые); нотация `(0,0)=a8`
- Русские шашки: обязательное взятие, множественные прыжки, дамки на любое расстояние
- AI: `AIEngine.find_move()` → `_search_best_move()` (iterative deepening) → `_alphabeta()` (TT + LMR + killers) → `_quiescence()` → `_evaluate_fast()`
- Оценка: материал + продвижение + центр + связанные пешки + фазовые веса + расстояние дамки + золотые углы
- Детекция повторов и ничейных эндшпилей

## Версионирование

Следуем [Semantic Versioning](https://semver.org/). Полная история изменений — в **CHANGELOG.md**.

- Единственный source of truth: `draughts/__init__.py` → `__version__`
- **Коммиты в dev — версию НЕ меняют** (это рабочий процесс)
- **Версия меняется только при merge в master** (по запросу пользователя)
- Claude **самостоятельно определяет** номер версии по характеру изменений:
  - **Patch** (3.2.x): баг-фиксы, мелкие правки, косметика — ничего нового для пользователя
  - **Minor** (3.x.0): новые фичи, улучшения AI, новые настройки, новые CLI-параметры
  - **Major** (x.0.0): ломающие изменения (несовместимость save-файлов, смена архитектуры) — только по явному запросу пользователя
- При обновлении версии — **обязательно обновить CHANGELOG.md** (добавить новую секцию)
